## 1. 前端赛事 AI 页面接入 RAG 功能

### 问题描述

前端两个 AI 助手组件（独立页面 `src/views/front/AIAssistant.vue` 和导航栏弹窗 `src/components/AIAssistant.vue`）使用硬编码关键词匹配逻辑回答问题，需要接入后端已完成的 RAG 知识库系统。

### 解决方案

#### 新建文件

**`src/composables/useRagChat.js`** — 共享 RAG 聊天 composable，包含：
- 会话管理（localStorage 持久化 sessionId）
- 消息列表状态管理
- 对话历史加载/清除
- 与后端 `/api/rag/query/` 接口对接

#### 修改文件

| 文件 | 改动 |
|------|------|
| `src/views/front/AIAssistant.vue` | 删除 ~220 行硬编码 `getAIAnswer` 函数，替换为 `useRagChat` composable |
| `src/components/AIAssistant.vue` | 同上，弹窗打开时懒初始化会话 |

#### 数据流

```
用户输入 → useRagChat.sendMessage()
  → POST /api/rag/query/ {query, session_id}
    → 后端: 问候语检测 → FQA 检索 → RAG 生成
    ← JSON {answer, session_id, source}
  → 显示答案
```

## 2. Milvus 混合检索稀疏向量为空导致报错

### 问题描述

调用 RAG 时，Milvus 报错：

```
MilvusException: (code=65535, message=fail to search on QueryNode 14: 
worker(14) query failed: Assert "size > 0" at ...Utils.h:227
=> Sparse row data should not be empty)
```

### 根因

bge-m3 嵌入模型对某些短中文查询（如"如何报名"）生成的稀疏向量（sparse vector）为空 `{}`，即没有任何非零值。当这个空稀疏向量传给 Milvus `hybrid_search` 做混合检索时，Milvus 要求稀疏查询向量至少有一维非零值，因此抛出断言错误。

### 解决方案

**文件**：`rag/core/vector_store.py` — `hybrid_search_with_rerank` 方法

在构建稀疏检索请求前，检查稀疏向量是否为空。若为空，降级为纯稠密向量检索（`client.search`），跳过混合检索。

```python
if not sparse_query_vector:
    self.logger.warning(f"稀疏向量为空，降级为纯稠密检索。query: {query[:50]}")
    results = self.client.search(
        collection_name=self.collection_name,
        data=[dense_query_vector],
        anns_field="dense_vector",
        limit=top_k,
        ...
    )[0]
else:
    # 正常混合检索
    results = self.client.hybrid_search(...)[0]
```

## 3. 后端服务在流式响应后静默崩溃

### 问题描述

每次调用 LLM（大模型）查询后，Django 开发服务器进程静默退出，终端回到 PowerShell 提示符，无任何 Python traceback 错误信息。FQA（BM25+MySQL）查询不受影响。

### 排查过程

1. **排除 StatReloader**：使用 `python manage.py runserver --noreload` 禁用文件监听后问题依旧。
2. **排除 GPU 内存溢出**：FQA 查询走纯 CPU 路径也复现崩溃。
3. **定位触发条件**：只在 LLM 流式调用完成后退出的请求才触发崩溃。

### 根因

三层问题叠加：

1. `call_llm()` 方法**每次调用都 `new OpenAI()`**，底层创建新的 `httpx` HTTP 连接池，用完即弃。
2. `StreamingHttpResponse` 生成器结束后，WSGI 调用 `response.close()` → `generator.close()` → `GeneratorExit` 传播到嵌套生成器链中。
3. Windows 上 `httpx` 连接池在 `GeneratorExit` 触发的清理流程中执行 socket 操作，导致进程静默退出。

### 解决方案

#### 修复 1：复用 OpenAI 客户端（减少连接池创建）

**文件**：`rag/qa_system.py` — `call_llm` 方法

```python
# 之前：每次 new 新客户端
client = OpenAI(api_key=..., base_url=...)

# 之后：复用 __init__ 时创建的单例
completion = self.client.chat.completions.create(...)
```

#### 修复 2：移除 StreamingHttpResponse（彻底避免生成器清理问题）

**文件**：`rag/views.py` — `RagQueryView`

将所有查询路径（问候语 → FQA → RAG）统一到 `/query/` 接口，内部收集完整 LLM 输出后一次性返回 JSON `{answer, session_id, source}`，不再使用 `StreamingHttpResponse`。

**文件**：`src/composables/useRagChat.js` — `sendMessage`

前端简化为只调用 `/query/` 一个接口，删除 SSE 流式解析逻辑。

#### 修复 3：增强异常保护

**文件**：`rag/views.py` — `_event_stream`（保留但不再被前端调用）

整个生成器体包裹在统一 `try/except` 中，任何异常都返回 SSE error 事件而非传播到 WSGI 层。

**文件**：`rag/qa_system.py` — `query` 方法

RAG 流式生成和 `save_conversation` 分别包裹在嵌套 `try/except` 中。

### 代价

RAG 回答不再逐字流式输出（用户需等 LLM 生成完才看到完整答案）。后续部署到生产 WSGI 服务器（gunicorn/uwsgi）时可恢复 SSE 流式。

## 4. FQA 精确匹配 + RAG 路径均未写入 Redis 缓存

### 问题描述

1. FQA 精确匹配路径：每次查询都查 MySQL，Redis 缓存始终为空。
2. RAG 流式生成路径：LLM 生成的答案也没有写入 Redis，同样问题重复走 RAG 全流程。

### 根因

`bm25_search.py` 的 `search()` 方法有两条返回路径缺失缓存回写：

| 路径 | 写缓存 |
|------|--------|
| ① 缓存命中 | 不需要 |
| ② **精确匹配** `query in self.origin_questions` | **缺失！** |
| ③ BM25 匹配得分超阈值 | ✅ |

`views.py` 的 `RagQueryView` 中 RAG 路径也没有缓存回写。

### 解决方案

#### 修复 1：FQA 精确匹配路径

**文件**：`rag/fqa/bm25_search.py` — `search()` 方法

```python
# 之前：查 MySQL 后直接返回
if query in self.origin_questions:
    return FqaEntry.objects.filter(...).first(), False

# 之后：查 MySQL 后回写 Redis
if query in self.origin_questions:
    answer = FqaEntry.objects.filter(...).first()
    if answer:
        cache.set(f"answer:{query}", answer, timeout=86400)
    return answer, False
```

#### 修复 2：RAG 路径

**文件**：`rag/views.py` — `RagQueryView.post()`

```python
# RAG 生成完成后，回写 Redis
if collected_answer:
    from django.core.cache import cache
    cache.set(f"answer:{query}", collected_answer, timeout=86400)
```

#### 缓存命中流程

```
用户提问 → bm25_search.search()
  → cache.get("answer:{query}")  ← 所有路径的答案都在这里命中
  → 有值？直接返回（秒级响应，无需 MySQL / LLM）
  → 无值？走 FQA → RAG → 写缓存
```

## 4. FQA 精确匹配路径未写入 Redis 缓存

### 问题描述

每次用户查询的问题返回答案后，Redis 缓存中没有任何数据。即使同一个问题反复查询，每次都要查 MySQL。

### 根因

`bm25_search.py` 的 `search()` 方法有三条返回路径：

| 路径 | 写缓存 |
|------|--------|
| ① 缓存命中 `cache.get(f"answer:{query}")` | 不需要 |
| ② **精确匹配 `query in self.origin_questions`** | **缺失！** |
| ③ BM25 匹配得分超阈值 | ✅ `cache.set(...)` |

路径②直接从 MySQL 取答案并返回，但**忘记调用 `cache.set()` 回写 Redis**。而日志显示绝大多数查询都命中路径②（精确匹配），所以缓存永远是空的。

### 解决方案

**文件**：`rag/fqa/bm25_search.py` — `search()` 方法第 129-132 行

```python
# 之前：查 MySQL 后直接返回，不写缓存
if query in self.origin_questions:
    return FqaEntry.objects.filter(question=query).values_list('answer', flat=True).first(), False

# 之后：查 MySQL 后回写 Redis，下次同样问题可从缓存命中
if query in self.origin_questions:
    answer = FqaEntry.objects.filter(question=query).values_list('answer', flat=True).first()
    if answer:
        cache.set(f"answer:{query}", answer, timeout=86400)
    return answer, False
```
